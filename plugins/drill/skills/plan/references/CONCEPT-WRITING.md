# Concept 작성 가이드 — KEEP/DROP, 좋은/나쁜 예, 자가 점검

> **단계별 질문 차원** — plan(spec/concept) = **what / why** / prepare(ticket) = **how** / write(코드) = how 따라 구현. 본 가이드는 plan 단계 산출물의 작성 기준이며, **how 어휘가 본문에 등장하면 ticket 영역으로 위임**한다.

## KEEP — 도메인 책임·약속·흐름

- 도메인 정의 (이 개념이 무엇인가)
- 도메인 책임 (responsibilities) · 도메인 규칙·정책 (business rules)
- 다른 도메인과의 관계 · 핵심 분류·상태 (도메인 어휘로)
- 사용자가 거치는 큰 단계 (추상 동사: "연결한다" / "해제한다" / "재연결한다")

## DROP — 본문에 등장하면 ticket / 디자인 영역으로 위임

| 카테고리 | 예시 (본문에 두지 않음) | 도메인 표현으로 |
|---------|-------------------------|------------------|
| 노출 매체·컴포넌트 | `Popover` / `Modal` / `Dialog` / `Toast` / `Sheet` / `Tooltip` / `AlertDialog` | "X 뷰" / "X 단계" / "X 안내" |
| 구현 동사 | `"뜬다"` / `"열린다"` / `"닫힌다"` | "확정된다" / "안내된다" / "정리된다" |
| 표준 프로토콜·시스템 어휘 | `OAuth` / `토큰` / `polling` / `BE` / `endpoint` / `필드명` | "외부 서비스 로그인" / "인증 만료" / "주기적 확인" / 시스템 주체 제거 |
| UI 위치·CTA 카피 | "리스트 상단 Callout" / "[가져오기]" 같은 정확한 버튼 텍스트 | 행동 의미만 ("다시 연결하기 동작") |
| API 시그니처·필드명·코드명 | `subscribeToTask` / `KnowledgeConnection.status` | "재구독 호출" / 정책 자체만 도메인으로 |
| 책임 분리 이력 | "X 는 서버에서 내려온다" / "마이그레이션 불필요" | 결과만 ("자연 정리된다") |
| 추상 메타 | `"분류 축"` / `"직교"` / `"두 도메인 축"` | "필터" / "라이프사이클" 같은 구체 도메인 |
| 매체 분기 결정 | `"X 0건 → 별도 안내"` / `"변경 없음 → Y skip"` | 도메인 동작 자체가 바뀌는 게 아니면 X |
| **선택지·옵션 카탈로그** | `"옵션 A/B/C 중 본 사이클"` / `"size selector / navigator"` | "X 를 사용한다" 까지만 (우선순위는 ticket) |
| **위치 중복 결정** | `"X 와 Y 양쪽에서 제공"` / `"Callout 과 More 메뉴 중복"` | "X 영역에서 가능" 까지 (위치는 ticket) |
| **카탈로그 항목 구체 값** | `"Dropdown 항목 = ON/OFF"` / `"4-state"` / `"5 액션"` | 행동 약속만 (카탈로그는 ticket) |
| **시그니처·prop 분기 어휘** | `"discriminated union"` / `"variant prop"` | 도메인 분기("단건/벌크 시나리오") 까지 |
| **how 어휘** | `"어떻게 / 어디에 / 어떤 매체"` | what + why 만 (how 는 ticket) |
| 카탈로그성 표·ASCII 플로우·픽셀·색·아이콘 | 동작 모두 나열한 표 / `[1차 모달]→[연동하기]` | bullet 단계 · 한 줄 도메인 어휘 |

단, `"비동기"` 같이 사용자에게 자연스러운 어휘는 그대로.

## 보조 원칙

- **decision 본문 인라인 링크 금지** — 본문(개요·책임) 의 줄마다 `([Decision X](path))` 박지 않는다. 영향 받은 decision 은 **하단 §관련 Decision** 섹션의 표(날짜·요약·영향 부분) 로 모음. 본문 인라인 ≤ 1 회 권장 (한 책임 영역당). 누적 3회 이상이면 이관 신호.
- **서비스별 분기 위임** — 상위 concept 에 모든 서비스 나열 X. "서비스마다 다름. {svc} 는 [[integration-{name}]] 참고"
- **중복 작성 방지** — 같은 정책이 여러 concept 에 반복 X. "X 의 Y 는 [[A]] 참고" 단일 source
- **표가 자족적이면 표 외 부연 반복 금지** — 표 아래 부연 불릿 5개 누적되면 표 재설계 신호. 표 + 도메인 경계 한 줄까지
- **줄 수 상한 없음** — 도메인 본질이 큰 경우 자연스럽게 길어질 수 있음. 다만 verbose / 구현 디테일 / 중복 결과로 길어진 게 아닌지 점검

## 좋은/나쁜 예

### 도메인 분류 — 단일 source

🚫 매체 매트릭스 노출:
```
연결 상태는 LNB 의 서비스 아이템 우측 배지, 페이지 리스트 상단 Callout,
헤더 CTA 배지에 동시 표현된다.
| 분류 | LNB 배지 | Callout 메시지 |
| expired | 인증 만료 | "재연결이 필요해요" |
```

✅ 도메인 어휘 + 단일 source:
```
연결 상태는 3분류이며 단일 source 로 모든 진입점에서 동일 분류로 표현된다.
| 분류 | 의미 |
| connected | 토큰 유효 + 권한 정상 |
| expired | 토큰 만료 — 재연결 필요 |
```

### 동작 흐름 — 매체 빠진 도메인 동사

🚫 매체 + 구현 동사 + CTA 카피:
```
[1차 모달] → [연동하기] → OAuth → [2차 모달]
연결 설정 Popover 에서 [연결 해제] 메뉴를 선택하면 AlertDialog 가
열리고, 확인 시 success Toast 가 뜬다.
```

✅ 도메인 사건 단계:
```
- 최초 연동: 외부 서비스 로그인 → Admin 판정 → 사용자 확정 시점에 connection 생성
- 연결 설정 뷰에서 사용자가 연결 해제를 선택하면 확인 후 해제된다.
```

### 뷰의 책임 — "다루는 정보·동작"

🚫 컴포넌트 매트릭스:
```
| 분류 | LNB 배지 | Callout | CTA | Popover 내부 Callout |
| expired | 에러 dot | error variant | "연결 만료" | error + 재연결 |
```

✅ 매체 중립 명칭 + 정보·동작 분리:
```
### 연결 설정 뷰

연결 완료 후 사용자가 진입하는 연결 설정 뷰가 다루는 정보·동작:

**공통**
- 연결된 계정 정보
- 현재 연결 상태 (정상 / 인증 만료 / 권한 부족)
- 연결 해제

**서비스별 추가**
- Jira 사이트별 상태 — [[integration-jira]]

연결 만료 시 재연결 진입은 별도 책임으로 [[reconnect]] 참고.
```

### 약속까지만 — 선택은 ticket

🚫 선택지 우선순위 + 위치 결정 + 카탈로그:
```
지식 목록은 페이지네이션을 사용한다. Footer paginator 는
size selector / navigator / count 를 모두 노출한다 (본 사이클).

다운로드 액션은 본문 첨부 Callout 의 [다운로드] 버튼 +
사이드픽 More 메뉴 [다운로드] 항목 양쪽에서 제공한다.

AI 게시 셀의 Dropdown 항목은 ON / OFF 두 가지로 구성된다.

사이드픽 More 메뉴는 5 액션을 노출한다:
복제 / 그룹 옮기기 / 다운로드 / 공유 / 삭제.
```

✅ 약속 + 도메인 어휘:
```
- 지식 목록은 페이지네이션을 사용한다 — 본 사이클 포함 옵션은 ticket
- 파일 타입은 사이드픽 안에서 다운로드 가능 — 노출 위치는 ticket
- AI 게시 ON / OFF 를 제어할 수 있다 — 컨트롤 표현은 ticket
- 지식 문서·파일의 상세 화면에서 가능한 액션: AI 게시 / 그룹 옮기기 /
  다운로드(파일) / 삭제 / 메타 확인. 노출 위치·진입 매체는 ticket.
```

### Decision 위치 — 본문 인라인 vs §관련 Decision

🚫 본문 인라인 매트릭스:
```
- **문서**: 텍스트 기반 에디터 지원 ([Decision import-unified](path)).
  외부 파일은 시스템 판단으로 자동 귀속 ([Decision file-upload-catalog](path)).
- **파일**: 13종 카탈로그 ([Decision file-upload-catalog](path)).
  검증 정책은 100건·10MB ([Decision file-upload-validation](path)).
```

✅ 본문 단언 + §관련 Decision 표:
```
- **문서**: 텍스트 기반 에디터 지원. 외부 파일은 시스템 판단으로 자동 귀속
- **파일**: 원본 보존, 입력 카탈로그 13 종. 검증은 개수·용량·카탈로그 일치 3 축

## 관련 Decision

| 날짜 | Decision | 영향 부분 |
|------|----------|-----------|
| 2026-05-18 | [import-unified-to-file-upload](path) | §콘텐츠 타입 |
| 2026-05-18 | [file-upload-catalog-13-types](path) | §콘텐츠 타입 |
| 2026-05-18 | [file-upload-validation-toast-matrix](path) | §콘텐츠 타입 |
```

> **패턴 핵심**:
>
> 1. **매체 중립 명칭** — "{X} 뷰" / "{X} 단계" (Popover / Modal X)
> 2. **다루는 정보·동작** 으로 헤더링 (정보=명사, 동작=동사)
> 3. **단위 다른 항목은 그룹 분리** (공통 / 서비스별 추가)
> 4. **세부 플로우는 외부 위임** ("재연결은 [[reconnect]] 참고")
> 5. **약속까지만** ("X 를 할 수 있다") — 위치·매체·옵션·카탈로그는 ticket
> 6. **decision 은 §관련 Decision 표로 모음** — 본문 인라인 ≤ 1
> 7. **단일 source** — 같은 사실이 여러 곳에 표현되면 같은 분류·표현 사용

## Concept 분리 vs 통합

| 상황 | 결정 |
|------|------|
| 자기 책임 영역이 명확히 다르고 디테일이 크고, 독립적으로 읽힘 | **분리** |
| Provider/Subtype 차이만 | **통합** (sub-section 으로) |
| 한 도메인 흐름(연결→사용→표현) | **통합** (분리 시 흐름 흩어짐) |
| 본문이 다른 concept 본문을 여러 번 참조해야 자명 | **통합** 신호 |
| 어디에 정보가 있어야 할지 헷갈림 | **통합** ("분리 비용 > 가치") |

사례: **integration / integration-jira / integration-knowledge** → **integration** 1개 (4부 구조, Provider 차이는 sub-section) — Jira 만 검색·필터 제공 같은 정보가 어느 concept 에 있는지 헷갈리지 않게.

## 표현 정확성 함정

다음 표현은 사실 확인 후 정정:

- **`"X 진입점이 제거되었다"`** — 진짜 못 하게 됐는지 vs Y 로 통합/흡수됐는지 확인. 후자라면 `"X 는 Y 흐름으로 통합"`
- **`"X 가 폐기"`** / **`"X 미지원"`** — 진짜 사용자가 X 를 할 수 없는지 vs 다른 방식으로 같은 결과를 얻는지
- **`"매트릭스 적재 및 X"`** 같은 사내 시스템 어휘 — 사용자 약속(`"에이전트가 답변에 활용 가능"`)으로 치환

## 자가 점검 키워드

본문에 다음 단어가 등장하면 도메인 표현으로 다시 쓴다 (DROP 표 참고). 매체 변경(Modal → Popover 등) / 선택지 변경 / 위치 변경은 spec cascade 가 아니라 ticket 영역이다.

- 매체·매체 동사: `Popover` / `Modal` / `Dialog` / `Toast` / `Sheet` / `Tooltip` / `AlertDialog` / `"뜬다"` / `"열린다"`
- 구현·책임 분리 이력: `OAuth` / `토큰` / `polling` / `BE` / `필드명` / `"서버에서 내려온다"` / `"마이그레이션"`
- 추상 메타: `"분류 축"` / `"직교"` / `"두 도메인 축"`
- 선택지·위치·카탈로그·시그니처: `"옵션 A/B/C"` / `"본 사이클 포함"` / `"양쪽에서 제공"` / `"Dropdown 항목 = ON/OFF"` / `"discriminated union"`
- how 어휘: `"어떻게"` / `"어디에"` / `"어떤 매체"`
- 본문 인라인 decision 누적: 한 책임 영역에 `[Decision X](path)` 2 회 이상 → §관련 Decision 으로 이관
