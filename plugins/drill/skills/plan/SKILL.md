# Plan

심층 인터뷰를 통해 해결하려는 문제를 파악하고, 기능을 Concept 단위로 분해하여 Spec 문서를 작성합니다.

## 출력 구조

```
~/Projects/flex/til/spec/{feature-name}/
├── {FEATURE-NAME}.md    # Index — 문제 정의, 전체 동작, concept 링크
└── concepts/
    ├── {name}.md        # 개별 개념 명세
    └── ...
```

## Workflow

### Phase 1: 주제 수집

- 사용자로부터 인터뷰 주제를 텍스트, Linear URL, Notion URL, Figma URL 등으로 받습니다
- Linear URL → 이슈 상세 로드
- Notion URL → 문서 로드
- Figma URL → 디자인 문서 로드
- 주제가 명확하지 않으면 AskUserQuestion으로 명확히 합니다
- 기존 `~/Projects/flex/til/spec/` 디렉토리에 관련 스펙이 있는지 확인합니다 (Glob 사용)

### Phase 2: 문제 정의 인터뷰

**목표**: "왜 이 기능이 필요한가", "누구를 위한 것인가", "어떤 문제를 해결하는가"를 명확히 한다.

**질문 방향:**
- 현재 사용자가 겪고 있는 문제는 무엇인가?
- 이 문제로 인해 어떤 불편함이 발생하는가?
- 이 문제 해결을 통해 어떤 가치를 창출하려 하는가?
- 이 기능의 성공 기준은 무엇인가?

**질문 방식:**
- AskUserQuestion 툴 사용, 한국어
- 다양한 각도에서 문제를 탐색하는 후속 질문 생성
- 문제 정의가 명확해질 때까지 반복
- 답변에 따라 follow-up 질문 생성

### Phase 3: 도메인 심층 인터뷰

**목표**: 기능을 구성하는 도메인 개념, 관계, 정책, 사용자 플로우를 파악하고, 각 개념이 **`templates/CONCEPT.md`의 섹션(개요 / 책임 / 에지 케이스 / 관련 Concept)을 채울 수 있는 수준**으로 구체화한다.

**질문 카테고리 (concept 유형별):**

- **도메인 모델**: 핵심 엔티티, 속성, 타입 구분, 엔티티 간 관계
- **구조/분류**: 데이터의 계층 구조, 분류 체계, 그룹핑 방식
- **프로세스**: 사용자가 거치는 절차, 시작점, 분기, 완료 조건
- **정책/권한**: 접근 권한, 역할별 할 수 있는 것, 제약 조건
- **에지 케이스**: 예외 상황, 경계 조건, 빈 상태

**각 카테고리 내에서 반드시 잡아야 할 축:**

| 축 | CONCEPT 섹션 | 예시 질문 |
|----|-------------|-----------|
| **정체성** | 개요 | "이 개념은 무엇인가요? 한 문장 정의는?", "왜 별도 개념이어야 하나요?" |
| **책임 영역·경계** | 책임 | "무엇을 담당하고 무엇은 담당하지 않나요?" |
| **규칙/플로우** | 책임 | (모델/구조/정책) 불변 규칙 / (프로세스) 사용자 플로우 |
| **관계·차이** | 관련 Concept | "다른 개념과 어떻게 구분되나요?", "의존하거나 영향을 주는 개념은?" |
| **경계 상황** | 에지 케이스 | "예외 / 빈 상태 / 삭제 / 만료에서 어떻게 되나요?" |

**질문 가이드라인:**
- 명백하거나 피상적인 질문 금지
- **구현 상세가 아닌 개념과 정책에 집중**: 파일명 / 함수명 / 라이브러리 / 로딩·애니메이션 UI / 데이터 포맷 등은 묻지 않음 (티켓·구현 영역)
- 두 개념이 붙었다 떨어졌다 하면 결정적 차이를 집중 질문 ("폴더와 카테고리의 차이는 무엇인가?")
- "왜?"를 3번 이상 물어볼 수 있는 깊이의 질문

**좋은 질문 예시:**
- "이 두 개념(폴더와 카테고리)은 어떻게 다른 건가요? 각각의 역할이 뭔가요?"
- "서비스별 권한자와 전체 관리자의 권한 경계는 어디인가요?"
- "연동 데이터와 직접 생성한 콘텐츠는 동일한 모델인가요, 별개인가요?"
- "카테고리를 삭제하면 해당 카테고리가 붙어있던 지식은 어떻게 되나요?"

**피해야 할 질문:**
- "어떤 기술 스택을 사용할 건가요?"
- "컴포넌트 구조를 어떻게 설계하시겠습니까?"
- "API 엔드포인트는 어떻게 구성하나요?"
- "로딩/애니메이션/에러 UI는 어떻게 보일까요?" — **구현 티켓 영역**

### Phase 4: Concept 식별

인터뷰 답변을 분석하여 기능을 **도메인 개념(Concept)**으로 분해합니다.

**Concept 유형:**
- **모델**: 핵심 데이터 엔티티와 속성 (예: "지식 문서", "카테고리")
- **구조**: 데이터의 계층/분류 체계 (예: "폴더 구조")
- **프로세스**: 특정 목표를 달성하는 절차 (예: "커넥터 연결")
- **정책**: 권한, 제약, 규칙 (예: "연동 서비스 권한")

**Concept 분해 기준:**
- 하나의 Concept = 하나의 온전한 도메인 개념
- 개발 단위가 아닌 **도메인 단위**
- 각 Concept은 독립적으로 설명할 수 있어야 함
- 다른 Concept과의 관계가 명확해야 함

**식별 절차:**
1. 인터뷰 답변에서 도메인 개념 추출 (엔티티, 구조, 프로세스, 정책)
2. 유사한 개념을 클러스터링
3. 각 클러스터에 Concept 이름 부여
4. Concept 간 관계 파악 (의존, 소속, 참조)

### Phase 5: Concept 확인

식별된 Concept 목록을 사용자에게 제시하고 확인받습니다.

```
다음과 같이 Concept을 분류했습니다:

1. **knowledge-document** — 지식 모델 (콘텐츠 타입, 속성, 생명주기)
2. **connector** — 써드파티 연결 프로세스 (모달, OAuth)
3. **folder-structure** — 콘텐츠 분류 구조 (1 depth 폴더)
4. **category** — 지식 라벨링 체계 (AI 활용 기준)
5. **connector-permission** — 연동 서비스 권한 정책

수정하거나 합치고 싶은 Concept이 있나요?
```

- 사용자가 합치기/분리/이름 변경을 요청하면 반영
- 빠진 개념이 있는지 확인

### Phase 6: 문서 작성

확정된 Concept 목록으로 Spec 문서를 작성합니다.

**{FEATURE-NAME}.md** — `templates/SPEC.md` 형식 참조:
- 파일명은 feature name을 대문자 + 하이픈으로 (예: `STREAMING-ANIMATION.md`)
- 제목의 feature name은 **대문자**로 작성 (예: `# MARKDOWN ANIMATION - Spec`)
- 문제 정의 (Phase 2에서 수집)
- 전체 동작 설명
- Concepts 테이블 (각 concept 파일 링크)
- **Concept 연결**은 하이픈 리스트로 `[name](concepts/name.md) → [name](concepts/name.md): 설명` 형식. 코드블럭 안에 `[[concept]]`만 쓰면 GitHub/VS Code에서 링크가 렌더되지 않으므로 금지
- Decision Log (빈 테이블) — plan 단계에서는 엔트리를 채우지 않음. Decision Log는 review 단계(`/drill:review`)에서 Spec과 구현의 차이가 발견될 때만 작성
- 피그마/노션 참조 링크

**concepts/{name}.md** — `templates/CONCEPT.md` 형식 참조:
- 개요: 이 개념이 무엇인지
- 책임: 이 개념이 해야 할 일, 사용자 플로우, 에지 케이스. 다른 concept에 영향을 주는 부분은 `[[concept]]` 인라인 링크
- 관련 Concept 링크

**Concept 원칙:**
- **단일 책임**: 각 Concept은 자기 책임만 다룬다. 책임 외의 것은 진행하지 않는다
- **책임 확장**: 필요하면 책임을 키우고 개념도 변경한다
- **영향 명시**: 자기 동작이 다른 concept에 영향을 주면 `[[concept]]`로 표시
- **구현 상세 제외**: 파일 경로, 코드 스니펫, 함수명, 컴포넌트명 등
- **구체적 코드명 사용 금지**: `subscribeToTask` → "재구독 호출", `MessageList.tsx` → "메시지 리스트"
- 사용자 플로우는 ASCII 다이어그램으로 표현

**저장 위치:** `~/Projects/flex/til/spec/{feature-name}/` (프로젝트 루트 기준)

### Phase 7: 최종 확인

스펙 문서 작성 후:
- 사용자에게 스펙 문서 경로 안내
- Concept별 핵심 책임·에지 케이스 요약 제공
- 추가 인터뷰 필요 여부 확인
- 다음 단계 안내: `/drill:prepare`로 티켓 생성

## 인터뷰 진행 흐름

```
1. 주제 수집 (Linear/Notion/Figma/텍스트)
   ↓
2. 문제 정의 인터뷰
   ↓
3. 도메인 심층 인터뷰
   ├─ 도메인 모델/구조
   ├─ 프로세스/정책
   └─ 에지 케이스
   ↓
4. Concept 식별 (도메인 개념 분해)
   ↓
5. Concept 확인 (사용자 검증)
   ↓
6. 문서 작성 ({FEATURE-NAME}.md + concepts/*.md)
   ↓
7. 최종 확인
```

## 인터뷰 완료 판단

다음 조건이 모두 충족되면 인터뷰 종료:
- 핵심 도메인 개념이 식별됨 (모델, 구조, 프로세스, 정책)
- 각 Concept의 개요와 책임·에지 케이스가 정의됨
- Concept 간 관계가 파악됨
- 더 이상 중요한 미결 질문이 없음

## 중요 사항

- **언어**: 모든 질문과 커뮤니케이션은 한국어로 진행
- **개념 중심**: 구현이 아닌 개념에 집중. "어떻게 만들 것인가"가 아니라 "어떻게 동작해야 하는가"
- **연속성**: 답변에 따라 자연스럽게 이어지는 follow-up 질문 생성
- **유연성**: 사용자의 답변에 따라 질문 방향을 조정

## 에러 처리

- 사용자가 답변을 모르는 경우: 대안 제시 및 함께 고민
- 범위가 너무 넓어지는 경우: 우선순위 조정 제안, Concept 분리 검토
- Linear/Notion/Figma 접근 실패: 텍스트 기반으로 진행
