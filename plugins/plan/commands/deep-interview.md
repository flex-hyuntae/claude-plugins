# Deep Interview

이 스킬은 기술 구현, UI/UX, 우려사항, 트레이드오프 등에 대한 심층 인터뷰를 진행합니다.

## Workflow

### 1. 인터뷰 주제 수집

인터뷰를 시작할 때:

```
- 사용자로부터 인터뷰 주제를 텍스트 또는 파일 형식으로 받습니다
- 주제가 명확하지 않으면 먼저 명확히 합니다
- 기존 관련 파일이 있는지 확인합니다 (Glob, Read 사용)
```

### 2. 심층 질문 생성 및 진행

**질문 가이드라인:**
- **모든 질문은 한국어로 작성**
- 명백하거나 피상적인 질문 금지
- 깊이 있고 구체적인 질문에 집중
- 다음 카테고리를 포괄:
  - **기술 구현**: 아키텍처, 데이터 구조, 알고리즘, 성능 최적화
  - **UI/UX**: 사용자 플로우, 인터랙션 패턴, 접근성, 반응형 디자인
  - **우려사항**: 잠재적 문제점, 기술 부채, 보안, 확장성
  - **트레이드오프**: 대안 접근법, 장단점 비교, 결정 근거
  - **엣지 케이스**: 예외 상황, 에러 처리, 경계 조건
  - **의존성**: 외부 시스템, API, 라이브러리 연동
  - **테스트**: 테스트 전략, 커버리지, QA 관점
  - **유지보수**: 코드 가독성, 문서화, 향후 확장 가능성

**질문 방식:**
- AskUserQuestion 툴을 사용하여 질문
- 한 번에 1-2개의 관련된 질문만 진행
- 답변에 따라 follow-up 질문 생성
- multiSelect 옵션을 적절히 활용

**질문 예시:**

피해야 할 피상적 질문:
- "어떤 기술 스택을 사용할 건가요?"
- "UI는 어떻게 디자인할 건가요?"

좋은 심층 질문:
- "사용자가 폼 작성 중 네트워크가 끊겼을 때, 작성 중이던 데이터를 어떻게 보존하고 복구할 계획인가요? localStorage vs IndexedDB vs 서버 임시 저장 중 어떤 방식이 적합할까요?"
- "1000개 이상의 아이템을 가진 리스트를 렌더링할 때, virtualization을 적용할 건가요? 적용한다면 어떤 라이브러리를 사용하고, 스크롤 위치 복원은 어떻게 처리할 건가요?"
- "모달 내부에서 또 다른 모달을 띄워야 하는 경우, z-index 관리와 포커스 트랩을 어떻게 처리할 건가요?"

### 3. 인터뷰 진행 흐름

```
1. 초기 질문 (기본 컨텍스트 수집)
   ↓
2. 기술 구현 심화 질문
   ↓
3. UI/UX 디테일 질문
   ↓
4. 우려사항 및 리스크 질문
   ↓
5. 트레이드오프 및 대안 검토
   ↓
6. 마무리 및 누락 사항 확인
   ↓
7. 스펙 문서 작성
```

### 4. 인터뷰 완료 판단

다음 조건이 모두 충족되면 인터뷰 종료:
- 모든 주요 카테고리에 대해 충분한 정보 수집
- 구현에 필요한 구체적인 결정사항 확보
- 우려사항과 해결 방안 명확화
- 더 이상 중요한 미결 질문이 없음

### 5. 스펙 문서 작성

인터뷰 완료 후 다음 형식의 스펙 문서를 작성:

```markdown
# [프로젝트/기능 이름] 기술 스펙

## 개요
[인터뷰를 통해 파악한 프로젝트/기능의 전체적인 설명]

## 요구사항
### 기능 요구사항
- [기능 1]
- [기능 2]

### 비기능 요구사항
- 성능: [성능 관련 요구사항]
- 보안: [보안 관련 요구사항]
- 접근성: [접근성 관련 요구사항]

## 기술 구현

### 아키텍처
[시스템 아키텍처 설명]

### 데이터 구조
\`\`\`typescript
// 주요 타입 정의
\`\`\`

### 주요 컴포넌트
1. **[컴포넌트명]**
   - 역할: [설명]
   - Props: [설명]
   - 상태: [설명]

### API 연동
- Endpoint: [엔드포인트]
- Method: [HTTP 메서드]
- Request/Response: [구조]

## UI/UX 사양

### 사용자 플로우
1. [단계 1]
2. [단계 2]

### 화면 구성
[주요 화면 및 컴포넌트 설명]

### 인터랙션 패턴
[사용자 인터랙션 설명]

## 우려사항 및 해결방안

### [우려사항 1]
- **문제**: [설명]
- **해결방안**: [설명]
- **트레이드오프**: [설명]

## 엣지 케이스 처리
- **[케이스 1]**: [처리 방법]
- **[케이스 2]**: [처리 방법]

## 테스트 전략
- 단위 테스트: [범위 및 도구]
- 통합 테스트: [범위 및 도구]
- E2E 테스트: [범위 및 도구]

## 구현 계획
### Phase 1: [단계명]
- [ ] 작업 1
- [ ] 작업 2

### Phase 2: [단계명]
- [ ] 작업 1
- [ ] 작업 2

## 참고사항
[기타 참고할 만한 정보]
```

**스펙 문서 저장:**
- 파일명: `spec-[project-name]-[YYYYMMDD].md`
- 저장 위치: 현재 디렉토리 또는 사용자가 지정한 위치
- Write 툴을 사용하여 저장

### 6. 최종 확인

스펙 문서 작성 후:
- 사용자에게 스펙 문서 경로 안내
- 주요 결정사항 요약 제공
- 추가 인터뷰 필요 여부 확인

## 인터뷰 팁

### 효과적인 질문 기법
1. **개방형 질문**: "어떻게 ~할 계획인가요?"
2. **구체적 시나리오**: "만약 ~한 상황이라면?"
3. **비교 질문**: "A 방식과 B 방식 중 어떤 것을 선택하시겠습니까? 그 이유는?"
4. **제약 조건 탐색**: "~한 제약이 있다면 어떻게 대응하시겠습니까?"

### 질문 깊이 확보
- "왜?"를 3번 이상 물어볼 수 있는 질문
- 실제 구현 코드를 상상하게 만드는 질문
- 여러 선택지가 있는 경우 각각의 장단점 비교 요청

### 한국어 질문 예시
- "대용량 데이터를 처리할 때 페이지네이션과 무한 스크롤 중 어떤 방식을 선택하시겠습니까? 각 방식의 성능 영향과 UX 관점에서의 장단점을 어떻게 보시나요?"
- "여러 사용자가 동시에 같은 문서를 수정할 때 충돌을 어떻게 처리할 계획인가요? Operational Transform, CRDT, 또는 간단한 락 메커니즘 중 어떤 접근법을 고려하고 계신가요?"
- "모바일 환경에서 오프라인 동작이 필요한가요? 필요하다면 Service Worker와 IndexedDB를 사용한 오프라인 캐싱 전략을 어떻게 설계하시겠습니까?"

## 중요 사항

- **언어**: 모든 질문과 커뮤니케이션은 한국어로 진행
- **연속성**: 답변에 따라 자연스럽게 이어지는 follow-up 질문 생성
- **완성도**: 구현 가능한 수준의 구체적인 정보 수집
- **문서화**: 인터뷰 내용을 놓치지 않고 스펙 문서에 반영
- **유연성**: 사용자의 답변에 따라 질문 방향을 조정

## 에러 처리

인터뷰 중 문제 발생 시:
- 사용자가 답변을 모르는 경우: 대안 제시 및 함께 고민
- 범위가 너무 넓어지는 경우: 우선순위 조정 제안
- 기술적으로 불가능한 요구사항: 대안 제시 및 트레이드오프 논의
